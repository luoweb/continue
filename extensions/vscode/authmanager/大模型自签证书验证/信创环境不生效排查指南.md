# 信创环境下自签名证书配置不生效排查指南

在信创环境（如国产操作系统、国产 CPU）下，配置 `verifySsl: false` 不生效通常与 **底层网络库的证书加载机制**、**环境变量传递限制** 以及 **代理服务器干扰** 有关。

以下是信创环境下可能导致 `verifySsl: false` 不生效的几个核心原因及排查方案：

### 1. 底层 Node.js 的证书硬编码限制
信创操作系统（如统信 UOS、麒麟）通常会为了安全增强，对系统自带的 Node.js 或 VS Code 运行环境进行加固。

- **原因**：某些加固环境可能强制忽略了 `rejectUnauthorized: false` 这种降低安全性的代码调用。
- **排查**：检查控制台日志。如果 `Fetch agent options` 显示为 `false` 但请求依然报 SSL 错误，说明是底层运行时环境（Runtime）拦截了该操作。

### 2. 环境变量 `NODE_TLS_REJECT_UNAUTHORIZED` 被锁定
在信创环境中，系统可能会在全局配置文件（如 `/etc/environment` 或 `/etc/profile`）中强制设置了安全策略。

- **原因**：如果系统强制设置了 `NODE_TLS_REJECT_UNAUTHORIZED=1`，它可能会覆盖插件内部的设置。
- **解决**：尝试在启动 VS Code 前手动执行：
  ```bash
  export NODE_TLS_REJECT_UNAUTHORIZED=0
  code
  ```

### 3. 强制透明代理（Transparent Proxy）
信创网络环境下通常部署有深信服、奇安信等安全网关。

- **原因**：网关可能会对 HTTPS 流量进行“中间人解密（SSL Inspection）”。即使插件内部关闭了校验，但如果网关强制要求证书匹配，或者网关证书未安装在系统信任库中，连接仍会中断。
- **验证**：在浏览器中访问你的 `apiBase` 地址，点击小锁图标查看证书链。如果证书被网关替换（如颁发者变为网关品牌名），则说明存在干扰。

### 4. caBundlePath 路径格式不兼容
如果你使用了 `caBundlePath` 而非 `verifySsl: false`：

- **原因**：信创系统的路径分隔符与权限控制较为严格。
- **解决**：
  - 确保路径是绝对路径。
  - 确保文件对当前用户有读取权限。
  - 在 Linux 架构的信创系统中，确保路径不包含中文或特殊字符。

### 5. VS Code 版本的特殊性
某些信创定制版的 VS Code（基于开源版本二次开发）可能修改了网络层逻辑。

- **排查**：在开发者工具 Console 中，除了看 `Fetch agent options` ，还要看是否有 `net::ERR_CERT_AUTHORITY_INVALID`。如果出现此错误且 `verifySsl` 已经是 `false`，说明插件调用的 `https.Agent` 被环境层劫持了。

---

### 建议操作步骤：

1. **强制全局忽略**：在终端启动时输入 `export NODE_TLS_REJECT_UNAUTHORIZED=0`。
2. **抓包确认**：如果可能，使用 `curl -k` 测试。如果 `curl -k`（忽略证书）在信创环境下也报错，说明是网络层网关拦截。
3. **日志对比**：对比信创环境和普通 Windows 环境下 `Fetch agent options` 的输出是否有差异。
