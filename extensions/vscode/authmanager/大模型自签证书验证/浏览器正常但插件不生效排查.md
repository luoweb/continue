# 浏览器访问正常但插件不生效排查指南

如果浏览器可以正常访问（没有报安全警告），但插件仍然无法连接，这说明 **系统/浏览器的证书信任库** 与 **Node.js（VS Code 运行环境）的证书信任库** 不一致。

以下是针对“浏览器正常但插件报错”的精准排查与解决方法：

### 1. 核心原因：Node.js 不读取系统证书库
浏览器（如 Chrome、Edge）通常使用操作系统的证书管理中心。而 VS Code 插件运行在 Node.js 环境中， **Node.js 默认只信任内置的一组根证书** ，不一定会读取你手动安装到系统里的自签名证书。

### 2. 解决方法

#### 方案 A：在 config.yaml 中强制忽略（最快）
即便浏览器正常，Node.js 依然可能校验失败。通过以下设置跳过 Node.js 的严格校验：

```yaml
requestOptions:
  verifySsl: false
```

#### 方案 B：指定证书文件（最安全）
既然浏览器正常，说明你手头有证书。直接让插件读取它：

```yaml
requestOptions:
  caBundlePath: "C:\\path\\to\\your\\cert.pem"
```

#### 方案 C：使用环境变量（全局修复）
在信创环境或 Windows 终端启动 VS Code 前，设置以下变量。这会告诉 Node.js 显式忽略所有 SSL 校验错误：

- **Windows (PowerShell)** : `$env:NODE_TLS_REJECT_UNAUTHORIZED="0"; code`
- **Linux/信创 (Terminal)** : `export NODE_TLS_REJECT_UNAUTHORIZED=0 && code`

### 3. 如何验证“环境差异”？
你可以通过以下对比来确认问题点：

1. **检查控制台日志** ：打开开发者工具（ `Ctrl+Shift+P` -> `Toggle Developer Tools` ）并切换到 Console 标签，搜索 `Fetch agent options` 。
   - 如果 `verifySsl` 是 `true` ，且报错 `CERT_AUTHORITY_INVALID` ，说明 Node.js 确实没拿到你的证书。
2. **检查证书链** ：在浏览器访问接口，点击地址栏小锁 -> 证书信息 -> 证书路径。
   - 如果证书链的顶端是一个 **公司内部私有 CA** ，那么 Node.js 肯定不认识它，必须通过上述方案 A 或 B 处理。

### 总结
浏览器正常不代表 Node.js 正常。 在信创环境下，最稳妥的做法是使用 **方案 C** （环境变量）或 **方案 A** （ `verifySsl: false` ），因为这能绕过 Node.js 独立的证书校验逻辑。
